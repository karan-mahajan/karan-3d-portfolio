## Technique / recipe

The "container-collection mounting script" pattern:

- **`scenery.002` is a *container* collection**, not a content collection. It holds five child collections (linked in [013-collections.md](../01-foundation/013-collections.md)): `bridges`, `rocks`, `basaltRocks`, `slabs`, `road.001`.
- **Why the empty script?** Because `013_collections.py` only links collections *together* but doesn't mount any of them to the scene's master collection. Every top-level collection needs an explicit `bpy.context.scene.collection.children.link(coll)` call to be visible in the outliner / rendered.
- This script's only job is to **mount `scenery.002` at scene root** so its 5 children (`bridges`, `rocks`, `basaltRocks`, `slabs`, `road.001`) all become reachable.
- The pattern repeats for every other top-level "container" collection (`areas` mounts via its first child-section script, `terrain` mounts via 020, `grass` mounts via 021, etc.).

## Notable code patterns

- **"Mount-only" scripts**: this is the simplest script pattern in Bruno's pipeline (after `011_linestyles.py`). 12 lines that just ensure a collection sits at scene root.
- **`print('  added 0 objects')`**: the script honestly reports 0 objects — Bruno doesn't pretend it does more. The print line is generated by whatever export tool dumped this script.
- **Color tag** for this collection is set in 013-collections.py: `scenery.002 → COLOR_07` (red), marking it as a "misc / decoration container."
- **No `bpy.data.objects.new(...)` calls anywhere** — pure collection-mounting.
- **The dot-suffix in the name** (`.002`) is a Blender-default naming artifact: there's no `scenery.001` or `scenery` here, but `scenery` (no suffix) is a separate collection (used by the circuit area's `075_scenery.py`), and `scenery.001` is also separate (behind-the-scene's scenery).
