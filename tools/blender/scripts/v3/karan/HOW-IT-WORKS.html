How the terrain masks work

A plain-English walkthrough of what terrainWater, terrainGrass, terrainFurnitures, and "R/G/B channels" actually mean — and how a single 512×512 image controls grass placement, river depth, water color, and tile patterns all at once.

1. What is an "image channel"?

Every PNG / EXR image is really three grayscale images stacked together. Each one is called a channel:

  • R = the "red" grayscale — one number (0.0 to 1.0) per pixel
  • G = the "green" grayscale — one number (0.0 to 1.0) per pixel
  • B = the "blue" grayscale — one number (0.0 to 1.0) per pixel

When you see a color image on screen, you're seeing all 3 channels combined. But in Bruno's world, the "image" isn't shown to anyone — it's just a data table the shader looks up. He uses each channel to store a different piece of world information.

Key idea: Bruno uses image files as if they were spreadsheets. He doesn't care what the image looks like as a picture — he cares about the per-pixel numbers in each channel.

2. Bruno's three "control images"

Bruno has three 512×512 EXR files, each storing different world info:

terrainWater.exrWhere depressions are + where water shows
terrainGrass.exrWhere grass grows
terrainFurnitures.exrWhere tile/slab patterns appear (paths/walls)

Each image has 3 channels (R, G, B), so theoretically 9 separate masks are possible. In practice, Bruno only uses 4 of those 9:

3. What we've already authored (sections done so far)

terrainWater.R — we painted this in 02-ground-grass-base.py

For every pixel, we wrote a value 0.0 → 1.0 saying "how deep is the terrain here":

R = 0.0   → flat ground (no depression)
R = 0.65  → river bed (~1m deep)
R = 0.80  → small pond (~1.2m deep)
R = 1.00  → ocean edge / big pond (max depth -1.5m)

Bruno's Geometry Nodes modifier reads this channel and physically pushes the terrain mesh's Z down by R × -1.5m. That's how the river and ponds get their dip.

terrainWater.B — we painted this in 02-ground-grass-water.py

We made B = R (mirror), so water color appears exactly where depressions are:

B < 0.10           → no water tint (dry land)
B = 0.10 → 0.30   → terrain blends toward teal (shallow)
B = 0.30 → 1.00   → terrain blends toward navy (deep)

terrainGrass.G — NEXT thing we paint (this session)

This is what you asked about. Painting G = 0.7 at a pixel means:

  • The terrain shader colors that spot greenish (mixing toward the grass color)
  • The grass-scatter object grows blades at 70% scale at that spot

Painting G = 0 means: no green tint and no blades — you'll see the tan/sand base color (whatever RGB.001 is set to).

terrainFurnitures.R — we ZEROED this in 02-ground-grass-blank-bruno.py

This is the "tile/brick pattern" overlay you saw in Bruno's screenshot. Bruno used it for paths and floor patterns. We zeroed it because Bruno's path layout doesn't match karan's island — we'd author our own version when we do paths in a future sub-project.

4. How the layers stack in the terrain material

The terrain plane (one mesh: Plane.134) shows a color at each pixel. The shader computes it through a stack of mix nodes:

① base color = mix(tan, green, factor = terrainGrass.G)
② + dark-green deepening (factor = terrainGrass.G)
③ + teal water tint (factor = remap of terrainWater.B)
④ + navy water tint (factor = remap of terrainWater.B)
⑤ + tile pattern OVERRIDE (factor = terrainFurnitures.R)

Layer 5 (tiles) comes last, which is why Bruno's road network was cutting into our ponds — tiles override water. We fixed this by zeroing terrainFurnitures.

Then on top of the terrain plane, there's a separate object for blades:

⑥ grass blade triangles (Plane.003, scattered by Geometry Nodes)

Each blade's size = terrainGrass.G at that spot. So the same mask drives both the ground color (layer 1+2) and the visible blades (layer 6) — which is why everything visually lines up.

5. Why you see "orange shade under grass" in Bruno's screenshot

It's not a separate "shadow layer" — it's just layer ① showing through where Bruno set G to a value in between 0 and 1. Pure orange happens at G=0; pure green happens at G=1; somewhere in the middle gives you the orange-tinged-green look. The grass blades sit on top of that.

Karan's version: we'll set the "tan" color (layer ①'s left input) to a sand color from your reference image, and the "green" to the vibrant grass green we already sampled. Then when we paint terrainGrass.G, areas with G=0 read as sandy paths, areas with G=1 read as green meadow, and gradients give natural transitions.

6. The runtime "shadow at the blade root" (not visible in Blender)

Bruno also has a small extra trick in his Three.js runtime (Grass.js): each blade is drawn darker at its base than at its tip, proportional to terrainGrass.G. That's the "shadow under each blade" effect you see in his live website. Blender's preview doesn't show this — it's pure GPU shader code. We'll get it for free if/when karan's runtime ports over.
